  <action>
Replace the `FalkonSchemeReply::startPage()` method body (lines 151-172 approximately) with a two-stage cache version that prevents the private-browsing token substitution being baked into the static cache.

Current approach: single static QString `sPage` built once, `%PRIVATE-BROWSING%` substituted into it and cached forever.

Problem: if a non-private window caches the page first (empty private notice), a later private window in the same process receives the non-private cached version.

Fix — adopt the speeddialPage() two-stage pattern:

```
QString FalkonSchemeReply::startPage()
{
    // Stage 1: build the static template once.
    static QString dPage;
    if (dPage.isEmpty()) {
        dPage.append(QzTools::readAllFileContents(QSL(":html/start.html")));
        dPage.replace(QLatin1String("%TITLE%"), tr("New Tab"));
        dPage = QzTools::applyDirectionToPage(dPage);
    }

    // Stage 2: per-request copy with dynamic tokens.
    QString page = dPage;
    page.replace(QLatin1String("%PRIVATE-BROWSING%"),
        mApp->isPrivate()
            ? QSL("<div class=\"pm-private-notice\">%1</div>").arg(tr("Private Browsing"))
            : QString());
    return page;
}
```

Remove ALL of the following token replacements from the old method because the new `start.html` no longer uses them: `%ABOUT-IMG%`, `%ABOUT-IMG-DARK%`, `%BUTTON-LABEL%`, `%SEARCH-BY%`, `%WWW%`, `%ABOUT-FALKON%`.

Do not alter any other methods in falkonschemehandler.cpp (speeddialPage, aboutPage, restorePage, configPage, etc.).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTML/JS in QtWebEngine → QWebChannel → C++ | JS from `falkon:start` calls `qz_prometheus_start` bridge methods |
| `bridge.navigateTo(url)` | URL string from user omnibox input or bookmarks data propagates to C++ navigation |
| `bridge.openAgentWithPrompt(prompt)` | Natural language string from user typed input propagates to agent sidebar |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-02 | Elevation of Privilege | QWebChannel bridge exposed to all pages | mitigate | Bridge is URL-guarded in externaljsobject.cpp (Plan 03): only registered on `falkon:start`, not arbitrary pages |
| T-10-03 | Spoofing | Malicious page impersonating `falkon:start` to access bridge | mitigate | URL guard checks `page->url()` AND `page->requestedUrl()` in Plan 03; scheme handler only serves the HTML for `falkon:start` |
| T-10-04 | Tampering | `bridge.navigateTo(url)` navigating to dangerous internal URL | mitigate | In Plan 03 `navigateTo` implementation: validate URL is http/https only (`QUrl(url).scheme()` must be `http` or `https`); reject javascript: data: and falkon: URLs |
| T-10-05 | Information Disclosure | `bridge.openAgentWithPrompt` auto-submitting without user confirmation | accept | Spec requires prompt is NOT auto-submitted; implementation in Plan 03 only calls seedPromptAndOpen which sets text without triggering AgentRuntime::submitTask — no further mitigation needed |
| T-10-06 | Tampering | Private browsing token baked into static cache | mitigate | Two-stage cache in startPage() ensures %PRIVATE-BROWSING% is substituted per-request, not cached |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No package manager installs in this plan; all assets are Phase 9 qrc: resources |
